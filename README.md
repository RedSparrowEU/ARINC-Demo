# AeroNav Data Loader

AeroNav Data Loader is a portfolio-grade aviation data update demo built to show senior-level engineering judgment across Apple/mobile, desktop, file-system workflows, data validation, diagnostics, and cross-platform delivery.

The project contains two applications:

- `desktop/` — a macOS desktop application built with Electron, focused on package import, file validation, removable-media export simulation, diagnostics, and operator workflows.
- `mobile/` — an iOS application built with Swift and SwiftUI, focused on package status, manifest import, validation history, diagnostics review, and companion/operator workflows.

The project is intentionally not a certified aviation system. It uses public or mock data to simulate downstream workflows around aviation navigation-data packages: import, validation, checksum verification, compatibility checks, export preparation, and diagnostic reporting.

## Why this project exists

The goal is to demonstrate the engineering skills expected in an iOS/macOS platform role that involves more than screens and API calls:

- native iOS development with Swift and SwiftUI,
- macOS/desktop workflow implementation,
- file-system and removable-media handling,
- manifest parsing and validation,
- checksum verification,
- offline-first state and operation history,
- diagnostic report generation,
- clear architecture boundaries,
- automated tests around business-critical logic,
- practical AI-assisted development without hiding engineering responsibility.

## Domain model

The demo models a simplified aviation data update workflow.

A real-world aviation navigation-data update flow may involve:

```text
Official aeronautical information sources
↓
Certified or commercial data provider pipeline
↓
Device-specific navigation/chart/terrain/obstacle package
↓
Desktop update tool
↓
USB / SD / removable media
↓
Aircraft avionics / FMS / GPS unit
```

This project focuses only on the downstream client-side part:

```text
Import package
↓
Parse manifest
↓
Validate files and metadata
↓
Calculate and verify checksums
↓
Check compatibility with a target device profile
↓
Export to a selected folder as removable-media simulation
↓
Verify exported files
↓
Generate diagnostics
```

## Important aviation-data disclaimer

This project does not provide operational aviation data and must not be used for flight planning, aircraft operation, navigation, maintenance release, or avionics loading.

The project may use public FAA CIFP-style data or generated sample files for demonstration. Raw public navigation data still requires professional processing and validation before any operational use. This demo only shows software architecture and workflow handling.

## Product concept

AeroNav Data Loader contains two connected applications.

### Desktop app: AeroNav Update Console

The Electron/macOS app is the primary operator tool. It simulates the workflow used by a pilot, operator, maintenance engineer, or QA engineer to prepare an aviation data update package for a removable medium.

Core responsibilities:

- import a navigation-data package from a folder or archive,
- parse `manifest.json`,
- inspect package contents,
- calculate SHA-256 checksums,
- validate required files,
- validate AIRAC/effective-date metadata,
- validate target-device compatibility,
- export the package to a selected folder that represents USB/SD media,
- verify checksums after export,
- write `export-log.json`,
- generate `diagnostics-report.json`,
- persist operation history locally,
- expose failure cases for QA/lab validation.

### Mobile app: AeroNav Companion

The iOS Swift app is a companion/operator view. It does not attempt to perform the full desktop file-system workflow. Instead, it demonstrates native iOS implementation and focuses on portable review and validation workflows.

Core responsibilities:

- import or open a package manifest,
- display package metadata,
- display validation status,
- display package contents by data category,
- show validation history offline,
- review diagnostics generated by the desktop tool,
- optionally scan or import a short diagnostics summary.

## Device profiles

Device profiles define compatibility requirements for export.

Example:

```json
{
  "id": "FD1000",
  "name": "FlightDeck FD-1000",
  "mediaType": "SD_CARD",
  "requiredRootFolder": "AVIONICS/NAVDATA",
  "supportedRegions": ["USA", "EUR"],
  "requiredCategories": ["navigation"],
  "optionalCategories": ["charts", "terrain", "obstacles"],
  "requiresSignedPackages": true,
  "maxPackageSizeMb": 4096
}
```

The demo should not use real manufacturer names unless there is a clear legal and licensing reason to do so.

## Validation rules

The validation layer should be deterministic and testable.

Minimum validation checks:

- manifest exists,
- manifest is valid JSON,
- required manifest fields are present,
- package ID matches expected naming convention,
- effective dates are valid,
- package is not expired,
- all required files exist,
- each declared file is inside the package root,
- SHA-256 checksums match,
- target device exists,
- package region is supported by selected device,
- required categories are present,
- export target is writable,
- post-export checksums match source checksums.

Suggested validation states:

- `notImported`,
- `imported`,
- `validating`,
- `valid`,
- `warning`,
- `failed`,
- `exporting`,
- `exported`,
- `exportFailed`.

Suggested issue severity:

- `info`,
- `warning`,
- `error`,
- `blocking`.

## Desktop feature roadmap

### Phase 1 — Import and manifest validation

- Select package folder.
- Locate and parse `manifest.json`.
- Show package summary.
- Show package file tree.
- Calculate checksums.
- Produce validation report.

### Phase 2 — Device compatibility

- Add device-profile configuration.
- Select target device.
- Validate region/category/media compatibility.
- Show actionable compatibility errors.

### Phase 3 — Export workflow

- Select export target folder.
- Check write permissions.
- Create required target folder structure.
- Copy files.
- Generate `export-log.json`.
- Recalculate checksums after export.
- Show final export status.

### Phase 4 — Diagnostics and QA fixtures

- Generate `diagnostics-report.json`.
- Add sample packages for failure scenarios.
- Add tests for each failure scenario.
- Add import/export logs to operation history.

### Phase 5 — Route and chart preview

- Add simplified route preview from mock or derived index files.
- Add PDF chart preview where legally safe.
- Keep previews clearly labeled as non-operational demo visualizations.

## Mobile feature roadmap

### Phase 1 — Package list and details

- Display sample packages.
- Show package metadata.
- Show package status.
- Show package contents by category.

### Phase 2 — Manifest import

- Import `manifest.json` through iOS document picker.
- Parse with `Codable`.
- Validate required fields.
- Show validation result.

### Phase 3 — Offline history

- Store validation history locally.
- Show previous validations.
- Support empty/error/loading states.

### Phase 4 — Diagnostics review

- Import or open `diagnostics-report.json`.
- Display issue list.
- Group diagnostics by severity.
- Show suggested actions.

### Phase 5 — Optional companion bridge

- Read compact diagnostics summaries from QR code or text export.
- Keep the feature optional and clearly separated from core validation.

## Security and safety notes

- Never execute files from imported packages.
- Treat all imported package content as untrusted.
- Prevent path traversal during import and export.
- Verify that exported paths remain under the selected target folder.
- Avoid storing secrets in repository files.
- Use mock signing keys only for demo signature workflows.
- Mark all aviation sample data as non-operational.
- Do not include proprietary Jeppesen, Garmin, Collins, Honeywell, or other vendor data unless usage is explicitly licensed.

## Implementation status

Phase 1 is implemented for both applications:

- desktop selects a package folder, parses `manifest.json`, displays metadata and a file tree,
  calculates SHA-256 checksums, and produces a deterministic validation report;
- mobile displays representative sample packages and navigates to metadata, status, effective dates,
  and package contents grouped by category.

Desktop device compatibility/export and mobile document import/decoding begin in later phases. No
diagnostics, persistence, route preview, or operational aviation behavior is implemented.

Phase 2 is also implemented: desktop validates selectable fictional device profiles, while mobile
imports and validates manifest JSON through the iOS document picker. Desktop export and mobile
offline history remain later phases.
Phase 3 is implemented: desktop exports verified declared files and a versioned export log to a
selected folder, and mobile persists the newest 100 validation attempts in an atomic local store.
Phase 4 is implemented: desktop emits v1 diagnostics and keeps operation history, while mobile imports and reviews diagnostics by severity.
Phase 5 is implemented: desktop provides non-operational route/PDF previews and compact bridge text, while mobile reads that text manually or by QR code on supported devices.
